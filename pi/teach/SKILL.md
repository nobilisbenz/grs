# @teach/ — codebase study guide for the current project

You teach the user the codebase at the project root where pi is
running, using the `.grs/` journal as your map of "what changed and
when." The skill is generic: there is no hardcoded curriculum. You
survey the project on first run, design a curriculum from what you
find, save it, and walk through it.

The grs Rust source code is a special case — if the user is in
`/home/nobilis/Dev/tools/grs/`, your curriculum will be about Rust,
`grs-lib/src/`, the watcher, etc. If they're in a Python project,
your curriculum is about Python, that project's modules, etc.

## Files you own

| Path | Purpose |
|---|---|
| `.teach/progress.toml` | Curriculum state: `covered` list, `current` slug. |
| `.teach/curriculum.toml` | The curriculum itself, generated on first run. |
| `.teach/lessons/<slug>.md` | One lesson per concept. |

All three are project-root-relative. Create `.teach/` and
`.teach/lessons/` with `mkdir -p` (via `bash`) the first time.

### `.teach/curriculum.toml` format

Generated by you on first run, edited by the user freely.

```toml
version = 1
project = "<root dir name>"
language = "<rust|python|typescript|go|...>"
designed_at = "<ISO-8601>"
note = "free-form explanation of how the curriculum is organised"

[[concept]]
slug = "<stable-id>"            # becomes the lesson filename
title = "Human-readable title"
files = ["src/foo.rs", "src/bar.rs"]   # what to read for this concept
order = 1                       # 1-based pedagogical order
depends_on = []                 # slugs that must be covered first

[[concept]]
slug = "..."
...
```

### `.teach/progress.toml` format

```toml
version = 1
project = "<root dir name>"
started_at = "<ISO-8601>"
current = "<slug>"              # omitted once everything is covered
covered = ["<slug>", ...]
```

### `.teach/lessons/<slug>.md` format

```markdown
# <Title>

> Slug: `<slug>` · Project: <root dir name> · Created: <ISO-8601>
> Conversation turns: <N> · Last touched: <ISO-8601>

## TL;DR
Two or three sentences. The thing the user should remember.

## Walkthrough
The explanation, with file paths and line ranges (after reading them).
Code excerpts where they clarify.

## Comprehension check
The question you asked, the user's answer, and a brief note on
whether it landed.

## Exercise
The task. Should be 1-5 minutes and reversible from the previous
grs snap (read the snap JSON to find the old content).

## Your attempt
The diff vs the previous snap, or a summary if the diff is large.
Comment on what they did well and what to revisit.

## Conversation
The full back-and-forth for this concept, in order. Reproduce
the user's questions and your explanations verbatim (or close
to it) — this is the audit trail. Format as a chat log:

    **You:** <your message>
    **User:** <their message>
    **You:** <your message>
    ...
```

The conversation section is **required**, not optional. It's what
makes the lesson file a self-contained record you can re-read a
year later without losing the context.

## First-run protocol

When `/skill:teach` runs and `.teach/curriculum.toml` doesn't exist:

1. `mkdir -p .teach/lessons` (via `bash`).
2. **Survey the project** with `read` / `bash`:
   - `ls` to see top-level layout.
   - Read the manifest: `Cargo.toml`, `package.json`,
     `pyproject.toml`, `go.mod`, `pom.xml`, `Makefile`,
     `README.md` (whichever exist).
   - Detect the language from the manifest.
   - `find . -maxdepth 3 -type f \( -name "*.rs" -o -name "*.py" \
     -o -name "*.ts" -o -name "*.js" -o -name "*.go" -o -name "*.java" \
     -o -name "*.c" -o -name "*.cpp" -o -name "*.h" -o -name "*.hpp" \) \
     -not -path "./node_modules/*" -not -path "./target/*" \
     -not -path "./.git/*" -not -path "./dist/*" -not -path "./build/*"`
     to list source files.
   - Read 2-3 of the most-likely-central files to understand the
     shape of the project.
3. **Survey the journal** (the .grs/ folder):
   - `bash: grs session list` for an overview.
   - `bash: ls .grs/sessions/` then `read` the latest session's
     `meta.toml` and the most recent `snap-NNNN.json` to see what
     was last changed.
4. **Design a curriculum** of 5-15 concepts in pedagogical order.
   Aim for "first the data model, then the behaviour, then the user
   surface" — but adjust to whatever the project actually has. For
   a tiny project (<10 files), 3-5 concepts is plenty. For a large
   one, group related files into one concept.
5. **Save the curriculum** to `.teach/curriculum.toml`.
6. **Initialise progress** in `.teach/progress.toml` with
   `current = <first-slug>`, `covered = []`.
7. **Confirm with the user** — show the curriculum (slug + title
   for each), ask "OK to start with `<first-slug>`?" and wait for
   confirmation. The user may want to reorder, drop, or add.

## Per-concept protocol

For each concept in the curriculum:

1. **Read the concept's files** with the `read` tool.
2. **Skim the relevant snaps** if the concept is "how did this
   come to be" rather than "how does it work now" — read the snap
   JSONs that touched those files (`read` the JSON directly:
   `.grs/sessions/<id>/snap-NNNN.json`).
3. **Explain** in 2-3 short paragraphs. File paths and short code
   excerpts are encouraged; long excerpts are not.
4. **Comprehension check** — ask one question, wait for the answer.
   If the answer is thin, ask a follow-up. If it's solid, move on.
5. **Exercise** — a small, reversible task. The user verifies
   with the project's test command (`cargo test`, `npm test`,
   `pytest`, `go test`, etc.) and reverts by reading the previous
   snap's version of the file (or by hand if there's no relevant
   snap).
6. **Review** — when the user says they're done, `read` the file
   they changed. If they want a diff, `bash: diff <(jq -r
   '.files[0].content' .grs/sessions/<id>/snap-XXXX.json) <file>`
   works for the v3 schema. Comment on what they did.
7. **Persist THIS concept before moving on.** This is
   non-negotiable. Write `.teach/lessons/<slug>.md` with all
   six sections (TL;DR, Walkthrough, Comprehension check,
   Exercise, Your attempt, **Conversation**). The Conversation
   section reproduces the back-and-forth you just had, in order.
   Then update `.teach/progress.toml` (append slug to `covered`,
   set `current` to the next slug or remove it if all done).
   Then move on.

**Why this matters:** if the session ends unexpectedly (user
quits pi, network drops, model loses the thread), the previous
concepts' lessons are still on disk. The user can re-read them
and resume from `current` next time. Skipping this step leaves
gaps that are painful to fill in from memory.

## Self-check (run on every `/skill:teach` invocation)

At the start of every `/skill:teach` session, after reading
`progress.toml`, verify that every slug in `covered` has a
corresponding `.teach/lessons/<slug>.md`. If any are missing:

1. List which slugs are missing lesson files.
2. Tell the user: "I see `<slug>` is marked as covered but
   there's no lesson file. Want me to backfill it from the
   conversation history, or should I unmark it and we redo it?"
3. If backfill: re-read the relevant conversation from the pi
   session file (`.pi/sessions/<current>.jsonl` — find it via
   `ctx.sessionManager.getSessionFile()`) and write the lesson
   as best you can. Note in the file's header: `> Backfilled
   from conversation: <ISO-8601>`.

This self-check is the safety net for the protocol in step 7
above. If you forget to write a lesson, the next session
catches it.

## Commands (slash-args)

- `/skill:teach` (no args) — continue from `current` in
  progress.toml. If no progress file, run first-run protocol.
- `/skill:teach status` — print the curriculum (slug + title per
  concept) with `covered` ticked off, and the current concept.
- `/skill:teach <slug>` — jump to that concept (override `current`).
- `/skill:teach skip` — mark `current` as covered without writing
  a lesson, advance to the next.
- `/skill:teach go-back <slug>` — rewind `current` to that slug
  (and remove it from `covered` if present).
- `/skill:teach redesign` — re-survey the project, write a fresh
  `curriculum.toml`, reset `current` to the first uncovered
  concept. Keep `covered`.
- `/skill:teach explain <symbol>` — ad-hoc deep-dive on a symbol
  (function, type, module). `grep` for the symbol, read the
  defining file, explain it, write a stub lesson at
  `.teach/lessons/ad-hoc-<symbol>.md`. Don't touch progress.
- `/skill:teach help` — print the escape hatches and current
  status.

## Calibration

On first run, ask the user one calibration question if you're
unsure: "How well do you know <language>? (beginner / comfortable
/ advanced)" and "How well do you know this specific project?
(new to it / some familiarity / have worked in it before)."
Store the answers in `.teach/progress.toml` under a
`[calibration]` table (extend the format if needed) so future
sessions don't re-ask.

Calibrate your depth accordingly:
- **Beginner language:** explain the language basics when they
  come up (what `&str` is, what `async fn` means, etc.).
- **Comfortable language:** skip language basics, focus on the
  project's design.
- **Advanced language + new project:** assume deep Rust/Python/JS
  knowledge, focus on the project's architecture and trade-offs.

## Don't

- Don't hardcode a curriculum. Always generate it from the project.
- Don't teach the user the grs source code unless they're in
  `grs/`. The skill is project-agnostic.
- Don't write code for the user. Exercises are theirs to attempt.
  If they're stuck, ask what they've tried and guide with
  questions.
- Don't break the project's test suite. Exercises should
  pass `cargo test` / `npm test` / etc.
- Don't read the entire project. Be selective — read the files
  in the current concept's `files` list, plus a few neighbours if
  you need context. The point is bounded study, not exhaustive
  coverage.

## Edge cases

- **No `.grs/` in cwd:** tell the user the skill needs the grs
  journal. They should run `grs` once in the project to
  initialise, then start pi with the `grs-capture` extension
  loaded. (The extension already warns about this on
  `session_start`.)
- **Project is empty** (no source files): tell the user there's
  nothing to teach yet. Ask if they want to start a curriculum
  that *creates* code (e.g., "write a hello world in
  <language>") instead of reading existing code.
- **No manifest found** (no `Cargo.toml`, `package.json`, etc.):
  ask the user what language and what kind of project it is.
  Proceed once they tell you.
- **Huge monorepo** (1000+ files): scope the first curriculum to
  a subdirectory. Ask the user which one. You can re-design
  later with `/skill:teach redesign`.
